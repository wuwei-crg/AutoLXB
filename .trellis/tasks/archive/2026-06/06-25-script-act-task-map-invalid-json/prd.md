# bug: script act task map invalid json trace

## Goal

Investigate and fix the recurring `script act` failure where one executed step ends with `fsm_script_act_task_map_step_end` and an `IllegalStateException` chain containing `task_map_first_step_window_timeout`, `task_map_visual_error`, and `invalid_json:jsonparse error at pos=`.

## What I already know

* The failure appears during the `script act` phase after executing one step.
* Reported trace/error marker: `fsm_script_act_task_map_step_end`.
* Reported exception chain includes `task_map_first_step_window_timeout`, `task_map_visual_error`, and `invalid_json`.
* The same failure appears in many traces, so trace writing or trace payload serialization may be involved.

## Assumptions (temporary)

* The failure is backend/runtime behavior, not a frontend/user-facing UI change.
* The root cause may be malformed JSON emitted into a task-map trace, malformed model output consumed as JSON, or a trace reader interpreting partial/corrupt data.
* Existing traces in the repository or configured trace output paths may contain enough evidence to identify the malformed payload shape.

## Open Questions

* None currently blocking. Local inspection indicates the likely fix belongs in model-output JSON extraction for script-act task-map visual resolution, not in trace JSONL writing.

## Backend Behavior (evolving)

* Diagnose where `fsm_script_act_task_map_step_end`, `task_map_first_step_window_timeout`, `task_map_visual_error`, and `invalid_json` are produced.
* Identify whether the malformed JSON is generated by trace writing, model response handling, visual task-map extraction, or post-processing.
* Fix the source of malformed JSON or add narrowly scoped defensive handling if the malformed payload can come from external/model output.
* `CortexFsmEngine.executeTaskMapRoutingStep()` catches the resolver exception and writes it into `fsm_script_act_task_map_step_end.reason`.
* `CortexFsmEngine.resolveFirstTaskMapTapPointWithWindow()` wraps the last visual resolver failure as `task_map_first_step_window_timeout:<lastErr>` after the first-step window expires.
* `SemanticVisionStepResolver.parseResult()` currently calls `Json.parseObject(raw.trim())` directly, so model responses wrapped in markdown fences or explanatory text become `invalid_json`.
* `TraceLogger.event()` serializes trace rows through `Json.stringify()`, which escapes strings; the observed message is expected to be a trace field value, not evidence that the trace row itself is invalid JSON.
* Fix: `SemanticVisionStepResolver.parseResult()` now keeps strict JSON parsing first, then falls back to the existing `CortexLlmHelper.extractJsonObjectFromText()` extraction for model responses that contain a JSON object inside markdown fences or explanatory text.

## Frontend/User-Facing Interface (evolving)
